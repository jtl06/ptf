# perf-mystery Agent Guide

## Project purpose

`perf-mystery` is a Linux performance-debugging lab built around this learning
loop:

`hypothesis -> tool -> evidence -> diagnosis -> fix -> validation`

Each puzzle provides a deliberately slow program, an equivalent fixed version,
progressive hints, a lesson, a diagnosis quiz, and an answer explanation.

The MVP uses only Python's standard library plus system tools:

- Python 3
- GCC/G++
- Make
- `strace`
- Linux `perf`

## Main entry points

- `pmystery.py`: CLI, puzzle metadata, build logic, profiling commands, quizzes,
  and journal creation.
- `Makefile`: builds all puzzles through `pmystery.py`.
- `README.md`: user-facing setup, workflow, examples, and architecture notes.
- `puzzles/`: source code and lesson material.
- `build/`: generated binaries; ignored by Git.
- `runs/`: profiling evidence and lab notes; ignored by Git.

Supported CLI commands:

```text
./pmystery.py list
./pmystery.py build <id>
./pmystery.py run <id> <bad|fixed>
./pmystery.py compare <id>
./pmystery.py strace <id> <bad|fixed>
./pmystery.py perf-stat <id> <bad|fixed>
./pmystery.py perf-record <id> <bad|fixed>
./pmystery.py lesson <id>
./pmystery.py hint <id> [level]
./pmystery.py diagnose <id>
./pmystery.py journal <id> <bad|fixed>
./pmystery.py reveal <id>
```

Puzzle IDs accept `1`, `01`, or the full directory slug.

## Puzzles

### 01: Syscall Storm

- Bad: reads an 8 MiB deterministic text file one byte per `read()` call.
- Fixed: reads with a 64 KiB buffer.
- Evidence: `strace -c` reports roughly 8.4 million `read()` calls for bad.
- Input is generated by `input/make_input.py` and is not committed.

### 02: Cache Maze

- Bad: follows a deterministic randomized linked traversal through a large node
  vector.
- Fixed: accumulates equivalent values from contiguous storage.
- Evidence: lower IPC and worse cache/TLB behavior where PMU events are
  available; `perf record` should identify the traversal loop.

### 03: Branch Lottery

- Bad: branches on the high bit of deterministic pseudo-random values.
- Fixed: computes the same selection using a branchless mask.
- Evidence: compare `branches` and `branch-misses`.
- GCC-specific attributes prevent if-conversion in the bad loop. Other compilers
  may lower it differently.

## Correctness invariants

- Bad and fixed variants must print exactly the same result.
- `compare` treats differing output as an error.
- Benchmark work must contribute to the printed checksum.
- Input and pseudo-random generation must remain deterministic.
- Avoid undefined behavior.
- Keep bad runtimes roughly within 0.2 to 5 seconds when run without tracing.
- Preserve frame pointers so `perf record -g` can produce useful call stacks.

## Build and validation

Run:

```sh
make clean
make
./pmystery.py compare 01
./pmystery.py compare 02
./pmystery.py compare 03
```

The project was validated on macOS for compilation/basic behavior and on the
Linux x86 hosts `alcidae` and `aethia`.

Observed on `aethia`:

- Puzzle 01: approximately 4.54 s bad versus 0.003 s fixed.
- Puzzle 02: approximately 1.55 s bad versus 0.009 s fixed.
- Puzzle 03: approximately 0.49 s bad versus 0.08 s fixed.
- `strace 01 bad`: 8,388,610 `read()` calls.

Timings are machine-dependent and should not be asserted exactly in tests.

## Profiling constraints

`strace` and `perf` may be absent or restricted. Keep failures concise and
actionable rather than exposing Python tracebacks.

On `aethia`, `perf_event_paranoid` was `4`. `perf stat` created its evidence file
but reported hardware counters as unsupported, and `perf record` was denied.
The CLI warns when output contains `<not supported>` or `<not counted>`.

Do not assume PMU event availability is identical across Intel, AMD, Graviton 3,
and Graviton 4. Prefer portable event categories and document architecture,
kernel, compiler, and flags when comparing systems.

## Development guidelines

- Keep the CLI readable and standard-library only.
- Store puzzle-specific metadata and quiz answers in `PUZZLES` in `pmystery.py`.
- Add new puzzles using the existing directory layout and bad/fixed contract.
- Do not commit generated inputs, binaries, profiling data, or lab journals.
- Do not overwrite an existing journal.
- Delete stale profiling output before starting a new collection.
- Update the root README when adding commands or changing workflows.

Potential future work is an Arm Performix collector backend that produces the
same evidence-file and journal workflow with Arm-specific PMU guidance.

## Remote host note

Use `alcidae` or `aethia` for Linux validation. Do not use `cancun`.
